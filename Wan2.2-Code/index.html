0全景与导读

本文档帮助你系统读懂 Wan2.2 视频生成框架的源码。Wan2.2 是阿里 Wan 团队开源的大规模视频生成模型套件，核心是一个 Latent Diffusion + DiT（Diffusion Transformer）+ Flow Matching 的架构。我们以最易上手的 TI2V-5B（稠密 5B 参数、文/图生视频）为主线，串起所有模块。

一句话理解整个系统：
文本/图像 → 用 UMT5 把 prompt 编码成语义向量；（图生视频时）用 VAE 把首帧编码成 latent
▼
扩散去噪 → 从纯高斯噪声出发，DiT (WanModel) 在压缩后的 latent 空间里迭代去噪 50 步，文本作为 cross-attention 条件
▼
解码出片 → 用 VAE Decoder 把去噪完成的 latent 还原成像素视频，存成 mp4
为什么选 TI2V-5B 作为主线？
A14B 系列（t2v/i2v）是 MoE 双专家（high-noise + low-noise 两个 DiT，按时间步切换），结构更复杂；s2v/animate 还要叠加音频/姿态等额外模态。TI2V-5B 是单一稠密 DiT，一个模型走到底，逻辑最干净，且同时支持「纯文生视频 t2v」和「图生视频 i2v」两条路径，是理解整套框架的最佳切入点。读懂它后再看其它任务只是「加料」。
官方论文图 · Wan 整体架构：左侧 Wan-Encoder(VAE) 把视频压成 latent，中间 N×DiT Blocks 在扩散过程中以 Timestep 和 umT5 文本(Cross Attention)为条件去噪，右侧 Wan-Decoder 还原视频。下面图 1 是结合本仓库代码画的对应详图。
图 1 · 系统总览：文本(UMT5)与可选首帧(VAE)编码为条件 → DiT 在 latent 空间迭代去噪 50 步(每步含 CFG 双前向) → VAE 解码出片

["一段文字"] → context: [L_text, 4096]　（L_text≤512）
▼
② 算 latent 目标形状　把像素尺寸按 vae_stride 压缩

▼

▼
⑦ 存盘　save_video：反归一化 (−1,1)→(0,255)，imageio 写 mp4 @ 24fps
wan/textimage2video.py · WanTI2V.t2v() 去噪主循环（节选+注释）
# ② latent 目标形状：像素 → 压缩空间
target_shape = (self.vae.model.z_dim,             # 48
                (F - 1) // self.vae_stride[0] + 1,    # 时间: 31
                size[1] // self.vae_stride[1],         # 高:   44
                size[0] // self.vae_stride[2])         # 宽:   80

# ③ DiT 序列长度（patch 后 token 数，向 sp_size 对齐）
seq_len = math.ceil((target_shape[2] * target_shape[3]) /
                    (self.patch_size[1] * self.patch_size[2]) *
                    target_shape[1] / self.sp_size) * self.sp_size

# ① 文本编码（条件 + 无条件/负向）
context      = self.text_encoder([input_prompt], self.device)
context_null = self.text_encoder([n_prompt], self.device)

# ④ 高斯噪声
noise = [torch.randn(*target_shape, dtype=torch.float32, generator=seed_g, ...)]

# 调度器：50 步时间表
sample_scheduler = FlowUniPCMultistepScheduler(num_train_timesteps=1000,
                                              shift=1, use_dynamic_shifting=False)
sample_scheduler.set_timesteps(sampling_steps, device=..., shift=shift)  # shift=5.0
timesteps = sample_scheduler.timesteps

latents = noise
for t in tqdm(timesteps):                  # ⑤ 50 步去噪
    # 给每个 latent token 一个时间步（TI2V 特有：逐 token 时间，支持图生视频混合）
    timestep = _expand_timestep_per_token(t, mask2, seq_len)

    noise_pred_cond   = self.model([latents[0]], t=timestep, context=context,      seq_len=seq_len)[0]
    noise_pred_uncond = self.model([latents[0]], t=timestep, context=context_null, seq_len=seq_len)[0]
    # Classifier-Free Guidance：放大「条件 − 无条件」的方向
    noise_pred = noise_pred_uncond + guide_scale * (noise_pred_cond - noise_pred_uncond)

    # 调度器推进一步
    temp_x0 = sample_scheduler.step(noise_pred.unsqueeze(0), t,
                                     latents[0].unsqueeze(0), return_dict=False)[0]
    latents = [temp_x0.squeeze(0)]

videos = self.vae.decode(latents)        # ⑥ 解码回像素
图生视频 i2v 的差异（同一函数体里）： ① 用 best_output_size() 把输入图按目标面积裁成 VAE 友好尺寸；② 用 z = self.vae.encode([img]) 把首帧编码成 latent；③ 用 masks_like(..., zero=True) 造一个「首帧位置=0、其余=1」的 mask，每一步去噪后都执行 latent = (1−mask)*z + mask*latent —— 即把首帧 latent 反复钉回去，只让模型生成后续帧。这就是「图像作为第一帧条件」的实现。

这一步的精髓：CFG（Classifier-Free Guidance）

模型每步预测两个噪声：一个「看到了 prompt」（cond），一个「看的是负向/空 prompt」（uncond）。最终用 uncond + s·(cond−uncond) 外推——s=guide_scale=5.0 越大越贴合 prompt 但可能过饱和。代价是每步要前向两次 DiT，这是推理时间主要来源。

def __call__(self, texts, device):
    ids, mask = self.tokenizer(texts, return_mask=True, add_special_tokens=True)
    seq_lens = mask.gt(0).sum(dim=1).long()        # 每条文本真实长度
    context = self.model(ids, mask)                  # [B, 512, 4096]
    return [u[:v] for u, v in zip(context, seq_lens)]  # 去掉 padding，返回 [L_i, 4096] 列表
UMT5 用的是 相对位置编码（T5RelativeEmbedding，按 bucket 给注意力加 bias），不带缩放的注意力（attn = q·kᵀ + bias，T5 风格），LayerNorm 是 RMSNorm 变体。这些是 T5 家族的经典设计，Wan 直接复用。

注意 --t5_cpu 可让这个 11GB 的编码器常驻 CPU，省显存；--t5_fsdp 在多卡下把它分片。编码完成后若 offload_model 开启，会立即 .cpu() 腾出显存给 DiT。

3DiT 主干 WanModel 核心

文件：wan/modules/model.py（546 行，整个框架的心脏）。这是一个 Diffusion Transformer：输入「带噪 latent + 时间步 + 文本」，输出「预测的噪声/速度场」。所有任务共用此类，靠 model_type 和超参区分。

💡 点击图中任意模块，下方展开对应源码（含 patch_embedding ↔ Head+unpatchify 的「沙漏」还原细节）。

图 3 · DiT (WanModel) 数据流（输入端口已按 WanAttentionBlock.forward 精确匹配）：带噪 latent x 经 patch_embedding 成为 token，作 Self-Attention 的 Q/K/V；文本 context 经 text_embedding 后只作 Cross-Attention 的 K/V；时间步 t 经 MLP 生成 6 路调制，仅作用于 Self-Attn 与 FFN 的归一化（橙色虚线），不进入 Cross-Attention。最终 Head+unpatchify 还原成与输入同形的速度场。
AdaLN 调制（图中橙色虚线）到底在做什么？ 它是把「扩散时间步 t」这一条件信号注入每个 Block 的方式——让同一套共享权重在不同噪声水平下表现出不同的去噪行为。三个目的：
  • ① 告诉模型「现在噪声有多大」。 去噪 50 步里，t≈1000（近纯噪声）与 t≈0（近干净）需要完全不同的预测策略。t 经 time_embedding → time_projection 变成 6 路向量 (shift, scale, gate)×2，逐 Block 注入，让模型按当前步调整行为。
  • ② FiLM 式条件注入，比拼接更省。 不把 t 拼到 token 上，而是用 t 去调制归一化后的激活：norm(x)·(1+scale)+shift。这是逐通道的仿射变换，参数少、算力省，却能在每层灵活地缩放/平移特征分布。
  • ③ gate 门控 + 末端归零保稳定。 残差分支输出再乘一个 gate：x = x + gate·y，让每层能自适应地缩放自身贡献。注意：经典 DiT 的「AdaLN-Zero」是把各 Block 的 gate 投影置零（初始恒等）；Wan2.2 并未这样做——它的 modulation/time_projection 都是常规随机初始化（model.py:217/386/532），gate 初始并不为 0。Wan 改为只把最终输出层 head.head.weight 置零（model.py:546），同样让训练初期网络输出≈0、保持稳定，但机制是「末端归零」而非「逐 Block 恒等」。
对应代码：e[0..2]=(shift,scale,gate) 调制 Self-Attn，e[3..5] 调制 FFN；Cross-Attention 不吃调制——它的条件来自文本 K/V，无需时间步信号。
WanModel 的四大组件：
组件作用形状变化
3 类 Embedding把 latent/时间/文本投到 dim=3072见 3.1
30 × AttentionBlock自注意力(时空) + 交叉注意力(文本) + FFN，全部由时间步「调制」[B, L, 3072] 不变
Head最后归一化 + 线性投影回 patch[B, L, 48·prod(patch)]
unpatchify把 token 序列还原成 latent 立方体→ [48, 31, 44, 80]
为什么输出 noise_pred 又和输入 latent x 同形？——DiT 是个「沙漏」。
patch_embedding 确实改变了形状（下采样 + 升维），但网络末端的 Head + unpatchify 是它的精确逆操作，一压一还原，形状回到原点。关键常被忽略的一点：Head 的输出维度不是 48，而是 48×prod(patch_size)=48×(1·2·2)=192——每个 token 吐出的是「它当初代表的那个 1×2×2 小块里、全部 48 通道的数值」，unpatchify 再把这 192 个数拆回小块、贴回空间位置。
图 3b · 为什么 noise_pred 与 latent x 同形：DiT 两端互逆，中段恒形。Head 投到 192=48×(1·2·2) 而非 48 是还原成立的关键——点上方图 3 的 Head 模块可看 unpatchify 的张量重排代码。

3.1 三类 Embedding

model.py · __init__ 中的 embedding 定义
# A. patch_embedding：3D 卷积把 latent 切块，等价于 ViT 的 patchify
self.patch_embedding = nn.Conv3d(in_dim, dim, kernel_size=patch_size, stride=patch_size)
        # in_dim=48 → dim=3072, kernel=stride=(1,2,2)

# B. text_embedding：把 4096 维文本投到 3072
self.text_embedding = nn.Sequential(nn.Linear(4096, dim), nn.GELU('tanh'), nn.Linear(dim, dim))

# C. time_embedding：正弦编码 → MLP；time_projection 再投到 6 份调制参数
self.time_embedding   = nn.Sequential(nn.Linear(freq_dim, dim), nn.SiLU(), nn.Linear(dim, dim))
self.time_projection  = nn.Sequential(nn.SiLU(), nn.Linear(dim, dim * 6))  # ★ ×6

「逐 token 时间步」是 TI2V 的关键设计。 普通 DiT 一个样本一个标量 t；这里 forward 接收的 t 形状是 [B, seq_len]——每个 latent token 可以有不同的时间步。这让「首帧已知(t=0)、其余帧待生成(t=T)」能在同一次前向里混合处理，正是图生视频的底层机制。时间嵌入因此产生 e0 形状 [B, L, 6, dim]。

3.2 AttentionBlock 与 AdaLN 调制

每个 Block 的结构是经典 DiT + AdaLN（自适应层归一化）：时间步通过 modulation 产生 6 个调制向量，控制自注意力、FFN 前后的「缩放/平移/门控」。注意：原始 DiT 论文的「AdaLN-Zero」会把各 Block 的 gate 投影零初始化使其起步为恒等映射；Wan2.2 并未这样做，它的 modulation 是常规随机初始化（model.py:217），仅把最终输出层 head.head 置零（model.py:546）来稳定训练初期——下文一律称其为 AdaLN 调制而非 AdaLN-Zero。

def unpatchify(self, x, grid_sizes):
    # ───── 入参维度与含义 ─────
    # x:          List[Tensor]，每个元素形状 [L, 192]
    #   L=27280   = patch 后的 token 数（带 padding；下面 u[:prod(v)] 会切掉 padding）
    #   192       = out_dim * prod(patch_size) = 48 * (1*2*2)，即「一个 token 打包的整块数值」
    # grid_sizes: Tensor [B, 3]，每行 = patchify 时 Conv3d 输出的网格尺寸 (F_p, H_p, W_p)
    #   本例 (31, 22, 40)：31 帧 latent、高 44÷2=22 块、宽 80÷2=40 块（去程记下，专供还原）
    # c = self.out_dim = 48（VAE z_dim，要还原出的通道数）
    c = self.out_dim
    out = []
    for u, v in zip(x, grid_sizes.tolist()):           # u:[L,192]  v=(31,22,40)
        # ① 切掉 padding，再把 192 这一维「拆开」成 patch 的内部结构
        #    prod(v)=31*22*40=27280 → 取前 27280 个有效 token
        #    view 把 [27280, 192] 重塑为 7 维 [F_p,H_p,W_p, p,q,r, c]
        u = u[:math.prod(v)].view(*v, *self.patch_size, c)  # →(31,22,40, 1,2,2, 48)
        #    f,h,w = 块坐标(31,22,40)；p,q,r = 块内偏移(1,2,2)；c=48 通道

        # ② 用 einsum 把「块坐标」和「块内偏移」交错排列，恢复像素的空间相邻顺序
        #    fhwpqrc → cfphqwr：通道c提到最前；高方向 h 与块内 q 相邻、宽 w 与 r 相邻
        u = torch.einsum('fhwpqrc->cfphqwr', u)        # →(48, 31,1, 22,2, 40,2)

        # ③ 合并相邻维度：H_p×p=22*2=44，W_p×r=40*2=80，F_p×1=31
        #    即把每个 2×2 小块「摊平」回它原来占的 2×2 像素位置 → 空间分辨率翻倍回去
        u = u.reshape(c, *[i*j for i,j in zip(v, self.patch_size)])  # →(48,31,44,80)
        out.append(u)
    return out                                       # List[Tensor]，每个 (48,31,44,80)
一句话理解 unpatchify：它是 patch_embedding 的精确逆操作。patchify 时每个 2×2 的空间小块被 Conv3d 压成 1 个 token（通道 48→3072）；unpatchify 时反过来——Head 先把每个 token 投回 192 = 48×(1×2×2) 个数（恰好是一个小块的全部像素值），再靠 grid_sizes 记下的网格尺寸 view + einsum + reshape 把这些数拆开、交错、贴回原来的空间位置，于是 22×40 的「块网格」展开回 44×80 的「像素网格」。
为什么需要 grid_sizes？ token 序列 [27280, 192] 是「拉平」的一维序列，本身不带空间结构。要把它折回 (F,H,W) 立方体，就必须知道当初是按什么网格拉平的——这就是 grid_sizes=(31,22,40) 的作用。它在 forward 里 patchify 之后立刻被 torch.stack([u.shape[2:] ...]) 记下来，一路透传到这里。u, v 只是 zip 的循环变量（按 batch 逐样本取「特征张量 u」和「它的网格尺寸 v」），与注意力的 q/k/v 无关。

输出形状回到了 latent 的 (48, 31, 44, 80)，与输入噪声一致——这正是去噪循环里 scheduler.step() 所需要的「预测」。

4.1 因果卷积 CausalConv3d

视频 VAE 必须保证「当前帧只依赖过去帧」（因果性），否则无法支持流式/分块处理。实现方式是只在时间维左侧 padding：

vae2_2.py · CausalConv3d
class CausalConv3d(nn.Conv3d):
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._padding = (W, W, H, H, 2*pad_t, 0)  # ★ 时间维只往「过去」pad，右侧=0
        self.padding = (0, 0, 0)

    def forward(self, x, cache_x=None):
        if cache_x is not None:           # 把上一块的尾帧拼到当前块前面
            x = torch.cat([cache_x, x], dim=2)
            padding[4] -= cache_x.shape[2]
        x = F.pad(x, padding)
        return super().forward(x)

4.2 分块编解码 + 特征缓存

为了省显存，VAE 把视频沿时间维分块处理（encode 每 4 帧一块，decode 每 1 个 latent 帧一块），块间用 feat_cache 传递边界帧的卷积中间结果，保证拼接处无缝。

vae2_2.py · WanVAE_.encode（分块循环）
def encode(self, x, scale):
    self.clear_cache()                       # 重置每个 CausalConv3d 的缓存槽
    x = patchify(x, patch_size=2)           # 2×2 像素打包 → 12 通道
    t = x.shape[2];  iter_ = 1 + (t-1)//4     # 按 4 帧分块
    for i in range(iter_):
        out_i = self.encoder(x[:,:, 块i, :,:], feat_cache=self._enc_feat_map, ...)
        out = out_i if i==0 else torch.cat([out, out_i], 2)
    mu, log_var = self.conv1(out).chunk(2, dim=1)   # 高斯分布参数
    mu = (mu - scale[0]) * scale[1]               # ★ 用预存的 mean/std 归一化（48维）
    return mu                               # 推理只取均值，不采样
scale = [mean, 1/std]： Wan2_2_VAE 在 __init__ 里硬编码了 48 维 latent 各通道的均值和标准差（一长串常数）。encode 后做 (mu−mean)/std 把 latent 归一化到标准尺度（利于扩散），decode 前做逆变换。这是 LDM 里「latent 标准化」的标准做法。

编码器/解码器主体是 3D 残差网络：ResidualBlock（双 CausalConv3d + RMSNorm + SiLU）+ AttentionBlock（中间层单头自注意力）+ Resample（上/下采样，含 AvgDown3D/DupUp3D 的 shortcut）。结构对称，encoder 下采样、decoder 上采样。

5流匹配调度器

文件：wan/utils/fm_solvers_unipc.py（UniPC）和 fm_solvers.py（DPM++）。Wan2.2 用的是 Flow Matching 框架（不是 DDPM），调度器负责给出时间步表、并在每步把 latent 往「干净」方向推进。

Flow Matching 直觉： 不像 DDPM 那样预测「加了多少噪声」，FM 学习的是从噪声分布到数据分布的 速度场（velocity）。sigma 从 1（纯噪声）线性降到 0（干净），timestep = sigma × 1000。
fm_solvers_unipc.py · set_timesteps 里的 shift（关键超参）

shift 的作用： shift=5.0 会把更多采样步数分配到高噪声阶段（视频生成的「大结构」在这里成形）。文档里提到「生成 480p 时建议 shift=3.0」——分辨率越低，shift 应越小。UniPC 是一个多步预测-校正求解器，比朴素 Euler 收敛更快，50 步即可出高质量结果。

step() 每步做的事：用 DiT 给出的 noise_pred（速度场）和当前 sigma，外加历史几步的缓存（多步法），解出下一个更干净的 latent。CFG 之后的 noise_pred 就是喂给它的输入。

pred_res = torch.einsum("k,bkc...->bc...", rhos_p, D1s)
x_t = x_t_ - alpha_t * B_h * pred_res                         # = 主项 − 修正

其中 h = λ_t − λ_s0 是对数信噪比的步长，B_h（bh1/bh2）与 h_phi_* 来自 e^h−1 的级数展开——本质是把指数积分器的高阶项用历史输出近似出来。读代码时不必死磕这些系数公式，记住结构是「基础步 + 历史信息修正」即可。

图 D · 多步法的「滑动窗口 + P-C 交错」：每步 UniP 用最近 order 个历史 x₀ 高阶外推下一步；从 step 1 起，UniC 用刚算出的新 x₀ 回头修正上一步的结果——几乎零额外模型调用（不需再跑 DiT）。开头不足 2 步时自动降为 1 阶（预热），结尾同理（lower_order_final）。

③′ 「阶数 order」到底指什么？（理论补充）

前面反复出现「1 阶 / 2 阶 / 降阶」，这里的阶数（order）是数值积分的概念，不是神经网络的层数，也不是张量的维度。它衡量的是：这一步用了「过去几个点」的信息来近似 ODE 的解曲线——用得越多，对曲线弯曲的拟合越准，单步误差越小。

一句话类比（开车估计下一秒的位置）：
阶数用到的信息开车类比单步局部误差
1 阶
Euler / DDIM		只用当前点的斜率 m0只看「当前速度」，假设这一秒匀速直线O(h²) 大
2 阶
UniPC 默认		当前 + 上一步，估出斜率的变化(D1s)同时看「速度 + 加速度」，知道在拐弯O(h³) 小
k 阶最近 k 个历史输出看到更高次的「加加速度…」O(h^{k+1})
其中 h = λ_t − λ_s0 是这一步的步长（对数信噪比空间）。步数越少 → 每步 h 越大 → 高阶项的优势越关键，这正是 UniPC「50 步抵 Euler 上百步」的根本原因。
图 E · 阶数的几何含义：1 阶用单点切线直线外推（粉，大步长会偏离弯曲的真实解）；2 阶借「上一步」估出曲率，沿弧线逼近（绿）。代码里 D1s=(mi−m0)/rk 就是在算这个「斜率的变化量」。
那「定阶数」这行代码在做什么？ this_order = min(收尾降阶, lower_order_nums + 1) 取两个上限的较小值：
① 预热下限 lower_order_nums + 1：第 1 步手里只有 1 个历史输出，没有「上一步」可比，只能做 1 阶；攒到第 2 步才够 2 阶。
② 收尾下限 min(solver_order, 剩余步数)（lower_order_final=True）：临近 σ→0 时主动降阶，让最后几步平稳收敛、不被高阶外推「甩飞」。
中间的绝大多数步则跑满 solver_order=2。所以一次 50 步采样的阶数序列大致是 1 → 2 → 2 → … → 2 → 1。
展开：为什么不一味用更高阶（比如 4 阶）？

高阶只降低局部截断误差，但需要更多历史步「预热」、对数值噪声更敏感，且多步法在大步长下高阶项可能不稳定（震荡放大）。扩散采样里步长 h 本就不小，2 阶（UniPC 默认 solver_order=2）已是精度/稳定性/缓存成本的甜点；UniC 的「免费校正」又额外补了一档精度，所以实践中 2 阶 UniPC 就足够在 50 步出高质量结果。这也是它比同阶纯 Predictor 的 DPM++ 更稳的原因。

和 DPM++ 的关系： 同文件夹的 fm_solvers.py 是 FlowDPMSolverMultistepScheduler（sample_solver='dpm++' 时启用），同样是多步指数积分器，但只有 Predictor、没有 Corrector。UniPC 多了一步几乎免费的校正，通常同步数下质量更好，所以是 TI2V 的默认 'unipc'。
一图记住 UniPC 每步： DiT 输出 v → convert 成 x₀ 预测 → （非首步）UniC 校正上一步 → 推入历史缓存 → 决定本步阶数（预热/收尾降阶）→ UniP 高阶预测下一个 latent → step_index+1。整段不含任何可学习参数，纯数值积分。

6.1 CFG 引导代码详解

CFG = Classifier-Free Guidance（无分类器引导），是扩散模型推理阶段提升「prompt 贴合度」的标准技巧。它不是训练损失，而是采样时对模型输出做的一次外推。

wan/textimage2video.py:380-386 · CFG 三行核心
# 每步把 DiT 前向跑两次：一次「看到 prompt」，一次「看到负向/空 prompt」
noise_pred_cond   = self.model(latent_model_input, t=timestep, **arg_c)[0]     # 条件: context = 真实 prompt
noise_pred_uncond = self.model(latent_model_input, t=timestep, **arg_null)[0]  # 无条件: context_null = 负向 prompt

# ★ CFG 外推：沿「条件 − 无条件」方向放大 guide_scale 倍
noise_pred = noise_pred_uncond + guide_scale * (noise_pred_cond - noise_pred_uncond)
图 6 · CFG（Classifier-Free Guidance）机制：每个去噪步把 DiT 前向跑两次（条件 / 无条件），再沿两者之差外推 s 倍。这是「prompt 贴合度」的来源，也是推理耗时翻倍的原因

数学形式与直觉（详细理论可参考https://zhuanlan.zhihu.com/p/2010512517017904543）

设模型在时间步 t、给定条件 c 时的预测为 v_θ(x_t, t, c)，则 CFG 的引导预测为： 
v_guided = v_θ(x_t, t, ∅)  +  s · [ v_θ(x_t, t, c) − v_θ(x_t, t, ∅) ]
         └─ 无条件(空/负向) ─┘     └────── 条件相对无条件多出来的「方向」──────┘
                                  s = guide_scale (TI2V 默认 5.0)
  • s = 1：退化为普通条件采样（没有额外引导）。
  • s > 1：放大 prompt 的影响，越大越贴合文字，但过大会过饱和/失真。TI2V-5B 默认 5.0。
  • 「无条件」用的是负向 prompt（context_null = config 里那串中文负向提示词，如「过曝、模糊、畸形…」），所以 CFG 同时起到「靠近正向、远离负向」的双重作用。
代价：每步两次 DiT 前向。 这是推理耗时的主要来源——50 步实际是 100 次 DiT 前向。这也是为什么 offload_model 模式下会在两次前向之间 torch.cuda.empty_cache()（见 i2v 分支 583/587 行）。
A14B（MoE）的差异： guide_scale 是一个 (low, high) 元组。text2video.py:343 里按时间步选用：高噪声阶段用 guide_scale[1]（high-noise 专家），低噪声阶段用 guide_scale[0]。TI2V-5B 是稠密单模型，全程一个标量 5.0。

6.2 损失计算 （该仓库仅推理）

真实训练损失：Flow Matching MSE

从调度器能反推出训练目标。Wan2.2 用 Flow Matching，调度器里写死了 prediction_type="flow_prediction"：

fm_solvers_unipc.py:317-322 · 模型输出的含义（反推损失）
sigma_t = self.sigmas[self.step_index]
if self.config.prediction_type == "flow_prediction":
    x0_pred = sample - sigma_t * model_output     # ★ model_output 是「速度场 v」

由 x0 = x_t − σ_t·v 反推：模型输出的 v 就是从干净数据指向噪声的速度场。对应的训练损失是标准的 Flow Matching（Rectified Flow）MSE：

# 训练时（不在本仓库，仅示意）：
x_0   = VAE.encode(真实视频)              # 干净 latent
noise = randn_like(x_0)                   # 高斯噪声
σ_t   ~ 采样一个 (0,1) 的噪声水平
x_t   = (1 − σ_t) · x_0  +  σ_t · noise   # 线性插值构造带噪样本

v_target = noise − x_0                    # 真实速度场（从 x_0 指向 noise）
v_pred   = WanModel(x_t, t, context)      # 模型预测

L = ‖ v_pred − v_target ‖²               # ★ Flow Matching MSE 损失
推理与训练的对应关系： 推理时调度器拿到的 noise_pred（CFG 之后的 v_guided）就扮演训练时 v_pred 的角色——只不过推理时不再算 loss，而是用 x0_pred = x_t − σ_t·v 把 latent 往干净方向推进一步（UniPC step()）。理解了这一点，训练和推理就闭环了。

7A14B 系列：MoE 双专家 进阶

前面用 TI2V-5B 把整套框架讲透了。t2v-A14B 在主干上完全复用同一个 WanModel，只是把它的「容量」和「调度方式」升级成了 MoE 双专家。本章只讲它与主线的差异点——读懂差异即读懂 A14B。

三句话概括 A14B 的不同：
  1. 更大的稠密 DiT：dim=5120, num_layers=40, num_heads=40（TI2V 是 3072/30/24）。
  2. 两个 DiT 专家：high_noise_model + low_noise_model，按扩散时间步在 boundary 处切换——这就是「27B 总参 / 14B 激活」的 MoE 含义。
  3. 用 Wan2.1 VAE（压缩比 4×8×8，比 TI2V 的 4×16×16 低），所以 latent 更大、in_dim=16 而非 48。

def _prepare_model_for_timestep(self, t, boundary, offload_model):
    if t.item() >= boundary:                 # 高噪声阶段
        required_model_name = 'high_noise_model'
        offload_model_name  = 'low_noise_model'
    else:                                    # 低噪声阶段
        required_model_name = 'low_noise_model'
        offload_model_name  = 'high_noise_model'
    if offload_model or self.init_on_cpu:        # 把不用的专家挪到 CPU，用到的挪到 GPU
        getattr(self, offload_model_name).to('cpu')
        getattr(self, required_model_name).to(self.device)
    return getattr(self, required_model_name)
